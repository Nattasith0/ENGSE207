# การสะท้อนคิด สัปดาห์ที่ 4: Layered Architecture

## 1. สิ่งที่เรียนรู้

[อธิบายสิ่งที่คุณเรียนรู้เกี่ยวกับ layered architecture]
Layered Architecture มีจำนวนไฟล์และบรรทัดโค้ดมากกว่า Monolithic เนื่องจากมีการแยกความรับผิดชอบของโค้ดออกเป็นแต่ละชั้น (Layer) เช่น Controller, Service และ Data Access ทำให้โค้ดไม่ถูกรวมอยู่ในไฟล์เดียว

## 2. ข้อดีที่พบจากการทำจริง

[บอกข้อดีที่คุณสังเกตเห็นขณะทำ lab นี้]
Layered เหมาะกับงานที่ต้องดูแลระยะยาว ขยายระบบ และทำงานเป็นทีม

## 3. ความท้าทายที่พบ

[อธิบายความท้าทายและวิธีแก้ปัญหา]
ความท้าทายหลักคือความซับซ้อนที่เพิ่มขึ้นในช่วงแรก นอกจากนี้ยังพบปัญหาเรื่อง route ไม่ตรงกับ frontend แต่สามารถแก้ไขได้ด้วยการไล่ตรวจสอบ flow การทำงานจาก Controller ไปยัง Service และ Database อย่างเป็นลำดับ

## 4. การจัดโครงสร้างโค้ด

การแบ่ง layers ช่วยให้การจัดโครงสร้างโค้ดดีขึ้นอย่างไรเมื่อเทียบกับสัปดาห์ที่ 3?

[ช่วยให้โครงสร้างโค้ดดีขึ้นอย่างมากเมื่อเทียบกับสัปดาห์ที่ 3 เนื่องจากโค้ดแต่ละส่วนมีหน้าที่ชัดเจน ไม่ปะปนกัน ทำให้โค้ดอ่านง่าย ดูแลง่าย]

## 5. เมื่อไหร่ควรใช้ Layered Architecture

[อธิบายสถานการณ์ที่ layered architecture เหมาะสม]
Layered Architecture เหมาะสำหรับโปรเจกต์ขนาดกลางถึงใหญ่ หรือโปรเจกต์ที่ต้องมีการพัฒนาต่อเนื่องในระยะยาว รวมถึงกรณีที่มีการทำงานเป็นทีมหลายคน เพราะสามารถแบ่งงานตาม layer

## 6. การวิเคราะห์ Trade-offs

### ข้อดี
- โค้ดมีความเป็นระเบียบและดูแลง่าย
-เหมาะกับการทำงานเป็นทีม

### ข้อเสีย
-โครงสร้างซับซ้อนกว่า Monolithic
-ใช้เวลาทำความเข้าใจและตั้งค่าเริ่มต้นมากกว่า

### การประเมินโดยรวม
[โดยรวมแล้ว Layered Architecture มีความคุ้มค่า แม้ว่าจะมีความซับซ้อนเพิ่มขึ้นในช่วงแรก แต่ช่วยให้การพัฒนาและบำรุงรักษาระบบในระยะยาวทำได้ง่ายและมีประสิทธิภาพมากขึ้น เหมาะสำหรับงานที่ต้องการความยืดหยุ่นและการขยายในอนาคต]
